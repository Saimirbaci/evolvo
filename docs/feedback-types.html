Feedback Types

How each type chosen in the Feedback panel shapes what the lineage agent does.

When a user submits feedback, Evolvo writes a FeedbackRecord and enqueues a LineageJobRecord. When the reviewer clicks Evolve, the runner builds a prompt from prompts/implementation.md and spawns the selected agent (Claude Code, Codex, Gemini, or OpenCode) inside an isolated git worktree. The feedback_type travels through to the prompt as a field and — for NewApp only — switches the runner into a different pipeline.

Defined in app/src-tauri/src/types.rs and mirrored in app/ui/src/types.rs. Order in the UI is controlled by FeedbackType::all().

The six types

New App special pipeline

Intent: "Start over — build me a different app on top of the Evolvo shell."

What the agent does:

  • The runner flips into new_app mode (see runner.rs / stages.rs) and drives a multi-stage pipeline — triage → plan → implement → build — rather than the single-shot implementation step used for the other types.
  • The prompt includes new_app_banner.md and work_steps/new_app.md, which tell the agent to treat this as a bootstrap: rewrite app/ui/src/app.rs (the NewApp content area), keep shell.rs intact, and preserve the product invariants (Lineage, Feedback Overlay, Canvas overlay on every page, one FAB).
  • The canvas screenshot is treated as the primary intent signal — the agent is expected to Read it before planning.
  • Iteration latitude is reset; the iteration counter no longer softens the "you may rewrite" signal.

Pick it when: you want a fundamentally different application inside Evolvo, not a tweak to the current one.

Feature

Intent: "Add something that isn't there yet."

What the agent does:

  • Uses the standard implementation prompt with the iteration-appropriate work step (bootstrap ≤ iter 5, shaping ≤ 10, maturation > 10).
  • Is told it is free to add new files, new Tauri commands, new Leptos components. Expected to register new commands in lib.rs invoke_handler and mirror them in interop.rs.
  • Expected to add tests for the new behaviour and update CLAUDE.md / .claude/rules if surface area grew.

Pick it when: the app is missing a capability and you want it added.

Improvement

Intent: "This already works; make it better."

What the agent does:

  • Same single-shot implementation prompt as Feature, but with a narrower remit — refine existing behaviour, polish UX, tighten copy, tune performance, improve accessibility.
  • Agent is biased toward minimal-diff changes and is expected not to introduce new top-level features or rewrite modules wholesale.
  • Still runs the full build contract (cargo check --workspace, cargo test, wasm check) and verifies by running the app.

Pick it when: a thing exists and basically works, but a rough edge needs sanding.

Bug

Intent: "Something is broken; fix it."

What the agent does:

  • Same implementation prompt, but scoped to a root-cause fix — the agent is told to diagnose, then patch the smallest set of files that resolves the behaviour the user described.
  • Agent is discouraged from drive-by refactors or adjacent cleanup — a bug fix should not grow into a feature PR.
  • A regression test is expected when the bug is testable. The canvas screenshot and voice transcript are read as the authoritative description of the broken behaviour.

Pick it when: the app is doing the wrong thing.

Confusion

Intent: "I don't understand what this is / what to do here."

What the agent does:

  • Treats the feedback as a UX / information-architecture signal rather than a functional defect. The fix is usually labelling, copy, layout, empty-state text, or an inline hint — not a new feature and not a behaviour change.
  • Expected to preserve the underlying behaviour while making the screen self-explanatory at the annotated spot (route is recorded on the feedback so the agent knows which page the user was looking at).
  • Good candidate for small, focused diffs in app/ui/src/ view components.

Pick it when: the app isn't broken, but you couldn't tell what it wanted from you.

Compliment

Intent: "I like this — keep it."

What the agent does:

  • No code change is expected. The agent records the compliment as a lineage note and resolves the job, which pins the praised behaviour as a signal for future iterations ("do not regress this").
  • Serves as positive training data for the staff-product-manager agent when it reviews the iteration and proposes next steps.
  • If the agent does decide a small reinforcing change is warranted (e.g. extract the praised component so it's harder to lose), it still runs the full build contract before committing.

Pick it when: something delighted you and you want the next iteration to preserve it.

Where this lives in the code

  • app/src-tauri/src/types.rs — FeedbackType enum (wire type, snake_case on the JSON side).
  • app/ui/src/types.rs — UI mirror + label() + all() ordering used by the chip row.
  • app/ui/src/feedback_panel.rs — renders the Type chips from FeedbackType::all().
  • app/src-tauri/src/runner.rs — branches on FeedbackType::NewApp to switch pipeline and work step.
  • app/src-tauri/src/stages.rs — the multi-stage NewApp planner/implementer that the other types bypass.
  • prompts/implementation.md — the shared prompt template; the {feedback_type} placeholder is the value the agent sees.
